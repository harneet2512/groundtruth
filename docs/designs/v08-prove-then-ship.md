### 2. `groundtruth check-diff` CLI Command

**What it does:** Reads a git diff and returns obligations for changed symbols. Can be used as a pre-commit hook, CI check, or manual verification.

**Invocation:**
```bash
# Default: reads git diff of staged changes
groundtruth check-diff

# Explicit: check diff between refs
groundtruth check-diff --base main

# From stdin (for CI pipelines)
git diff main..HEAD | groundtruth check-diff --stdin

# JSON output for programmatic use
groundtruth check-diff --format json

# Pre-commit hook mode: exit code 1 if obligations found
groundtruth check-diff --strict
```

**Builds on:** `ObligationEngine.infer_from_patch(diff_text)` — already implemented. `_parse_changed_symbols(diff_text)` extracts function/class definitions from unified diffs using regex. This is NOT LSP-based — it works from the diff text directly.

**Implementation path:**
1. Add `check-diff` subcommand to `main.py` argument parser and dispatch handler in `_dispatch()`. Include `--root` (default: `os.getcwd()`), `--base`, `--stdin`, `--format` (text|json), `--strict` flags.
2. Read diff: if `--stdin`, read `sys.stdin.read()`; elif `--base` provided, `subprocess.run(["git", "diff", f"{args.base}..HEAD"], capture_output=True)`; else `subprocess.run(["git", "diff", "--staged"], capture_output=True)`. Handle non-zero git exit codes.
3. Load index from `--root` path (same pattern as other CLI commands: `SymbolStore(db_path)`)
4. Call `ObligationEngine.infer_from_patch(diff_text)`
5. Format output as text (default) or JSON
6. Exit code: without `--strict`, always 0. With `--strict`, exit 1 when obligations > 0.

**JSON output schema:**
```json
{
  "obligations": [
    {
      "kind": "constructor_symmetry",
      "source": "UserModel.__init__",
      "target": "UserModel.deconstruct",
      "target_file": "src/models/user.py",
      "target_line": 45,
      "reason": "deconstruct references {name, email} but misses {phone}",
      "confidence": 0.85
    }
  ],
  "total": 2,
  "files_changed": 3
}
```

**Expected text output:**
```
$ groundtruth check-diff --base main

2 obligations found in 3 changed files:

  constructor_symmetry (confidence: 0.85)
    UserModel.__init__ → UserModel.deconstruct
    src/models/user.py:45
    Reason: deconstruct references {name, email} but misses {phone}

  override_contract (confidence: 0.90)
    BaseHandler.process → AdminHandler.process
    src/handlers/admin.py:23
    Reason: overrides BaseHandler.process — signature change propagates

Run `groundtruth check-diff --format json` for machine-readable output.
```

**Edge cases:**
- No git repo → error: "Not a git repository"
- No index exists → error: "No GroundTruth index found. Run `groundtruth index` first."
- Empty diff → "No changes detected."
- Changed files not in index → skip silently (file may be new/unindexed)
- Massive diff (10k+ lines) → process normally; `_parse_changed_symbols` is regex-based and fast

**Complexity note:** The diff-to-symbol mapping uses `_parse_changed_symbols()` which extracts `def foo` and `class Bar` from added `+` lines via regex. Known limitations:
- Won't catch changes to function bodies that don't modify the def line (false negative — safe)
- Deletions and renames are not detected — only added `def`/`class` lines. A renamed function will generate obligations for the new name but NOT for callers of the deleted old name. Note this in `--help` text.
- This is acceptable for v0.8. False negatives are safe (missed obligation). False positives would be bad but the regex approach doesn't produce them.

**Effort:** S (30 min CC). Core logic exists. New code: argparse subcommand, subprocess call, output formatting.

### 3. Django Coupling Demo in README

**What it does:** Run `groundtruth index` against a checkout of `django/django`, capture the enhanced summary output plus specific obligation examples, include in README.

**Pre-validation required:** Before writing the README section, actually run GT against Django locally. Verify:
- Index completes without errors (AST parser, no LSP required)
- Obligations fire correctly on Django's Field classes

**If Django indexing fails or produces poor results:** Fall back to a smaller, well-known repo (FastAPI, SQLAlchemy, or Requests). The point is real data from a real repo, not specifically Django.

**README section structure:**

**Timebox:** 30 minutes. If Django AST indexing does not complete cleanly within 30 min, switch immediately to FastAPI or Requests.

**Effort:** XS-S (15-30 min CC depending on whether Django indexes cleanly). Must prototype before committing.

## 12-Month Vision (NOT v0.8 scope)

This section describes the long-term direction. None of this is being built now.

### 10x Check
GroundTruth becomes the repository's memory — a living, evolving model that accumulates intelligence over time. Session memory tracks what agents get wrong. Convention inference learns the rules, not just the patterns. Agent-specific learning pre-empts failure modes per agent. Cross-repo intelligence transfers conventions across repos. Not a tool — an intelligence layer that gets better the more it's used.

### Platonic Ideal
Install GT with one command. GT silently indexes, infers conventions, maps coupling, identifies hotspots. First agent PR includes obligation checklist. Agent reads it, makes the changes, PR is correct on first try. User's reaction: "How did it know that?" Six months later, GT knows the repo better than any single engineer. Not a tool you call. Institutional knowledge, automated.

**Note:** The platonic ideal requires LSP servers to be pre-installed. v0.8 uses AST-only parsing (Python) which avoids this dependency. Multi-language support via LSP is a v1.0+ concern.
